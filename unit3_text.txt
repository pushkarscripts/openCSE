PART A - SEMANTIC ANALYSIS
1. INTRODUCTION TO SEMANTIC ANALYSIS
Semantic Analysis is the third major phase of a compiler, coming after Lexical Analysis and Syntax
Analysis. While Lexical Analysis deals with tokens (is each word valid?) and Syntax Analysis deals with
grammar (are words arranged in a grammatical structure?), Semantic Analysis deals with meaning --
does the program actually make sense logically and type-theoretically?
Consider this analogy: in English, the sentence "Colorless green ideas sleep furiously" is syntactically
correct but semantically nonsensical. Similarly, int x = "hello"; is syntactically correct C code but
semantically wrong because a string literal cannot be assigned to an int. Semantic analysis catches
exactly such errors.

1.2 What Does Semantic Analysis Check?
Semantic analysis is responsible for a wide range of correctness checks that go beyond pure syntax:
Type Checking:
Every operator must receive operands of compatible types. The compiler infers or checks types of every
expression.
Scope Checking / Name Resolution:
Every identifier must be declared before it is used. The compiler checks that a name is visible in the
current scope.
Arity Checking:
Function calls must supply the correct number of arguments matching the function's parameter list.
Flow-of-Control Checking:
break and continue statements must appear inside loops. return statements must match the function's
declared return type.
Initialization Checking:
Some compilers warn when a variable is used before being assigned a value.
Overloading Resolution:
In C++ and Java, the compiler uses argument types to determine which overloaded function version to
call.
KEY INSIGHT
Semantic analysis is the phase where the compiler builds most of the symbol table and uses it.
Unlike lexical and syntax errors which prevent further compilation, many semantic errors can be reported and
compilation can continue to find more errors.
2. ATTRIBUTE GRAMMARS - COMPLETE THEORY
An Attribute Grammar is the formal mathematical framework used to specify the semantics of a
programming language via a context-free grammar. The idea was introduced by Donald Knuth in 1968
and is the foundational formalism underlying all practical semantic specification in compilers today.
The core idea: each grammar symbol can have attributes -- named values associated with instances of
that symbol in the parse tree. Each grammar production has associated semantic rules that define how
attribute values are computed.
2.1 Formal Definition:
FORMAL DEFINITION
An Attribute Grammar is a triple AG = (G, A, R) where:
G = (N, T, P, S) is a context-free grammar with nonterminals N, terminals T, productions P, start symbol S.
A = {A(X) | X is a grammar symbol} is a finite set of attributes for each symbol.
R = {R(p) | p is a production in P} is a set of semantic rules for each production.
2.2 Types of Attributes - Synthesized vs Inherited:
Synthesized Attributes:
A synthesized attribute of a nonterminal A in a production A -> alpha is one whose value is computed
from attribute values of children of A in the parse tree, and/or attribute values of terminals in the
production body.
Think of it as information flowing upward through the parse tree -- from leaves toward the root.
Examples of synthesized attributes:
• E.val in an expression grammar (computed from children E1.val and T.val)
• E.type in a type-inference grammar (type of expression derived from operand types)
• T.place in a three-address code generator (temp variable name for this expression)
• digit.lexval (given directly by the lexer -- value of the digit token)
Inherited Attributes:
An inherited attribute of a nonterminal B at position j in a production A -> X1 X2 ... Xn is one whose
value is computed from attribute values of A (the parent), and/or attribute values of siblings to the LEFT
of B.
Think of it as information flowing downward or sideways -- from parent to children, or from left siblings
to right siblings.
Examples of inherited attributes:
• L.in in a declaration grammar (type inherited from D -> T L, where T.type flows to L.in)
• T.inh in a translation grammar (inheriting context from parent or left sibling)
• indent.level in a pretty-printer (current indentation depth passed downward)
• scope.level in a semantic analyzer (current nesting depth passed to all children)

Worked Example: Parsing and Evaluating 3 + 5 * 2
1. Lexer gives: digit(3), +, digit(5), *, digit(2)
2. F -> digit(3) : F.val = digit.lexval = 3
3. T -> F : T.val = F.val = 3
4. E -> T : E.val = T.val = 3 (left subtree done)
5. F -> digit(5) : F.val = 5
6. T1 -> F : T1.val = 5
7. F -> digit(2) : F.val = 2
8. T -> T1 * F : T.val = T1.val * F.val = 5 * 2 = 10
9. E -> E1 + T : E.val = E1.val + T.val = 3 + 10 = 13
10. L -> E newline : L.val = 13 -> print(13)
The final output is 13. Values are computed bottom-up -- leaves first, then their parents.

3. SYNTAX DIRECTED DEFINITIONS (SDD) - IN DEPTH
A Syntax Directed Definition (SDD) is a generalization of a context-free grammar in which each
grammar symbol has a set of attributes and each production has a set of semantic rules for computing
attribute values. An SDD is a pure specification -- it tells us WHAT to compute, but not HOW or in
WHAT ORDER to evaluate those rules.
FORMAL DEFINITION
An SDD consists of: (1) A CFG. (2) For each grammar symbol X, a set of attributes A(X). (3) For each production p:
A -> alpha, a set of semantic rules of the form b := f(c1, c2, ..., ck) where b is an attribute of some grammar symbol in
the production and c1..ck are attributes of grammar symbols in the production.
Restriction: if b is an attribute of A (LHS), it must be synthesized. If b is an attribute of a symbol in alpha (RHS), it
must be inherited.
3.1 SDD for Type Checking of Expressions:
Production

E -> E1 + E2
E.type = (E1.type==int AND E2.type==int) ? int : type_error

If any subexpression has type_error, the error propagates upward so that the entire expression is
flagged once.
4. SYNTAX DIRECTED TRANSLATION (SDT) SCHEMES - IN DEPTH
A Syntax Directed Translation Scheme (SDT) is an SDD with the semantic actions embedded directly
inside the production rule body using curly braces { }. SDT schemes are the bridge between
specification and implementation.
KEY DISTINCTION
SDD: Semantic rules listed separately from grammar. No execution order specified. Like a mathematical definition.
SDT: Semantic actions { } embedded at SPECIFIC POSITIONS inside production bodies. Position determines
WHEN actions run. Like an implementation recipe.
4.1 SDT for Infix to Postfix Translation:
E -> E1 + T { print('+') }
E -> T
T -> T1 * F { print('*') }
T -> F
F -> ( E )
F -> id { print(id.name) }
F -> num { print(num.val) }
Trace for input: a + b * c
Reduce F -> id(a) -> prints: 'a'
Reduce T -> F -> (no action)
Reduce F -> id(b) -> prints: 'b'
Reduce T1 -> F -> (no action)
Reduce F -> id(c) -> prints: 'c'
Reduce T -> T1*F -> prints: '*' output so far: a b c *
Reduce E -> E+T -> prints: '+' final output: a b c * +
Output: a b c * + which is the correct postfix notation.

5. S-ATTRIBUTED AND L-ATTRIBUTED DEFINITIONS
Two important subclasses -- S-attributed and L-attributed -- can be evaluated efficiently during a single
pass of parsing.
5.1 S-Attributed Definitions:
S-ATTRIBUTED DEFINITION
An SDD is S-attributed if it uses ONLY synthesized attributes.
Every semantic rule computes a synthesized attribute of the left-hand-side nonterminal.
No inherited attributes exist in the grammar.
S-attributed SDDs can be evaluated during a SINGLE BOTTOM-UP (LR) parse.
Evaluation order: POSTORDER traversal of the parse tree.
5.2 L-Attributed Definitions:
L-ATTRIBUTED DEFINITION
An SDD is L-attributed if for every production A -> X1X2...Xn, each inherited attribute of Xj depends only on: (1)
Inherited attributes of A (the parent), OR (2) Any attributes of X1, X2, ..., X_{j-1} (symbols to the LEFT of Xj in the
production body).
L-attributed includes ALL S-attributed SDDs as a special case.
L-attributed SDDs can be evaluated in a single LEFT-TO-RIGHT depth-first traversal.
Compatible with LL (top-down) parsers AND LR parsers.

6. EVALUATION ORDER - DEPENDENCY GRAPHS & TOPOLOGICAL SORT
Before we can evaluate attributes, we must determine a valid order of evaluation. We represent
dependencies formally using a Dependency Graph.
6.1 Dependency Graph Construction:
• Create a node in the dependency graph for each attribute instance in the parse tree
• For each semantic rule b := f(c1, c2, ..., ck), add a directed edge from each ci to b
• If the resulting dependency graph is a DAG, a valid evaluation order exists
• If the dependency graph has a CYCLE, the SDD is circular and CANNOT be evaluated

7. SYMBOL TABLE - COMPLETE THEORY
DEFINITION
A Symbol Table is a data structure created and maintained by the compiler to store information about every identifier
(variable name, function name, class name, etc.) encountered in the source program.
It is used and updated by virtually EVERY phase of the compiler -- the scanner creates initial entries, the parser adds
structural information, the semantic analyzer adds types and checks them, and the code generator uses it to allocate
memory and resolve addresses.

8. SYMBOL TABLE OPERATIONS
insert(name, attribute_list)
Adds a new identifier to the symbol table with its associated attributes.
• Called when parser/semantic analyzer encounters a declaration
• Must check for duplicate declarations in the same scope (error if found)
• Example: parsing 'int counter = 0;' inserts (counter, type=int, val=0, scope=current)
lookup(name)
Searches for an identifier and returns its symbol table entry (or null/error if not found).
• Called whenever an identifier is used in an expression or statement
• Must search from CURRENT (innermost) scope outward to global scope
• The most frequently called operation -- must be as fast as possible (hence hash tables)
delete(name) / delete_scope(scope_level)
Removes an identifier or all identifiers in a scope from the symbol table.
• Called when a block/scope is exited -- all local variables go out of scope
• In a stack-of-tables design, implemented by simply popping the top table
update(name, attribute, value)
Modifies an existing entry -- changes one or more attribute values.
• Used when new information is learned about an already-declared identifier
• Example: fill in the memory address/offset once memory layout is determined
9. DATA STRUCTURES FOR SYMBOL TABLE - ALL TYPES
9.1 Linear List:
The simplest implementation. Identifiers stored sequentially. Insertion appends to end; lookup scans
from beginning.
• Insert: O(1) -- append at end (but must check for duplicates: O(n))
• Lookup: O(n) -- linear scan -- very slow for large programs
• Suitable for: Very small programs, macro processors with fewer than 20 entries
9.2 Hash Table with Chaining (Most Common in Production Compilers):
A hash function maps an identifier name to an integer index (bucket). Each bucket is the head of a
linked list containing all entries that hash to that bucket. O(1) average-case performance for insert and
lookup.
// Simple hash function:
int hash(char *name, int size) {
int h = 0;
while (*name) h += *name++;
return h % size;
}
 
// Polynomial rolling hash (better distribution):
int hash(char *name, int size) {
int h = 0;
while (*name) h = (h * 31 + *name++) % size;
return h;
}
• Insert: O(1) amortized average -- compute hash, prepend to chain
• Lookup: O(1) amortized average -- O(n) worst case if all names collide
• Table size: Should be a prime number for better distribution (e.g., 211, 509, 1021)
• Used in: GCC, LLVM/Clang, Java VM, virtually all production compilers
9.3 Binary Search Tree (BST):
Identifiers stored in a BST keyed on name. O(log n) average for balanced trees.
• Insert/Lookup/Delete: O(log n) for balanced BST (AVL or Red-Black tree)
• Advantage: Inorder traversal gives identifiers in alphabetical order -- useful for cross-reference
listings
9.4 Stack of Symbol Tables (Block-Structured Languages):
For block-structured languages that support nested scopes, we maintain a stack of symbol tables --
one table per active scope level. Entering a block pushes a new table; exiting pops it.
int a = 1; // Table_0 (global): {a: int}
void f(int x) { // PUSH Table_1: {x:int}
int a = 10; // Table_1: {x:int, a:int} -- hides global a
{ // PUSH Table_2: {}
int b = 20; // Table_2: {b:int}
a = a + b; // lookup(a): Table_2 no, Table_1 YES -> local a
} // POP Table_2
} // POP Table_1

10. SCOPE MANAGEMENT IN DETAIL
Scope is the region of a program within which an identifier declaration is valid (visible).
10.1 Scope Rules:
• Static (Lexical) Scope: The scope of an identifier is determined by the textual structure of the
program. Used by C, Java, Python, Pascal, most modern languages.
• Dynamic Scope: The scope is determined at runtime based on the sequence of procedure calls.
Used historically by early Lisp dialects.
• Name hiding (shadowing): An inner declaration of the same name hides the outer declaration
within the inner scope.
10.2 Scope in C -- Detailed Rules:
• Global scope: declared outside all functions -- visible throughout the entire translation unit
• Block scope: variables declared inside {} -- visible from declaration to the closing }
• File scope with static: visible only in the current source file (no external linkage)

12. ACTIVATION RECORDS - ALL FIELDS EXPLAINED
When a procedure P is called, the runtime system allocates a block of memory on the stack called the
Activation Record (also called a Stack Frame). When P returns, its activation record is deallocated.
Return Value:
Space where P deposits its return value before returning. On some architectures, the return value is in a
register (EAX/RAX) rather than the stack.
• For small values (int, pointer): often in a register for efficiency
• For large values (structs): caller allocates space and passes pointer to it
Actual Parameters:
The values (or addresses) of arguments passed by the caller to P.
• Call by value: copies of values are stored here
• Call by reference: addresses of caller's variables are stored here
Control Link (Dynamic Link):
A pointer to the activation record of the CALLER. Used to restore the frame pointer (FP) when P returns.
• Always present in every activation record
• On P's return: FP = control_link (restores caller's frame pointer)
Access Link (Static Link):
A pointer to the activation record of P's lexical parent. Used to access non-local variables in nested
procedures.
• Only present in languages that allow nested procedure definitions (Pascal, Ada)
• Not present in C (which does not allow nested functions)
Saved Machine State:
Saved values of machine registers that P will overwrite but must restore before returning. Also includes
the saved PC (return address).
• Return address: placed by the CALL instruction; restored by RETURN instruction
• Callee-saved registers: P saves any register it uses that must be preserved
Local Variables:
Variables declared locally within P. Each has a fixed offset from the frame pointer, computed at compile
time.
• Only exist for the duration of P's activation
• Multiple recursive activations each have their own separate local variables
Temporaries:
Temporary values generated during expression evaluation that do not fit in registers.
• Compiler manages allocation of temporaries at compile time

13. PARAMETER PASSING MECHANISMS - ALL 4 TYPES
13.1 Call by Value:
CALL BY VALUE
The caller evaluates the actual parameter expression and passes only the RESULTING VALUE.
A fresh COPY of this value is created in the callee's activation record.
Any modifications to the formal parameter inside the procedure affect ONLY the local copy.
The caller's original variable is COMPLETELY UNAFFECTED.
void increment(int x) { // x is a copy of the actual parameter
x = x + 1; // modifies the LOCAL copy only
}
 
int main() {
int a = 10;
increment(a); // passes value 10; a copy is made
printf('%d', a); // prints 10 -- a is UNCHANGED!
}
• Used in: C (default for all types), Java (primitives)
• Advantages: Safe -- caller's data protected; simple to reason about
• Disadvantages: Expensive for large structures; cannot return multiple values through parameters
13.2 Call by Reference:
CALL BY REFERENCE
The caller passes the ADDRESS (L-value) of the actual parameter variable.
The formal parameter becomes an ALIAS (another name) for the caller's actual variable.
Any modification to the formal parameter DIRECTLY MODIFIES the caller's variable.
There is no copying -- both formal and actual refer to the same memory location.
void swap(int &x;, int &y;) { // x,y are references (aliases)
int temp = x;
x = y; // directly modifies caller's first variable
y = temp; // directly modifies caller's second variable
}
 
int main() {
int a = 5, b = 10;
swap(a, b);
printf('%d %d', a, b); // prints: 10 5 -- BOTH CHANGED!
}
• Used in: C++ (& parameters), Fortran (default), C# (ref keyword), Pascal (VAR parameters)
• Advantages: Efficient -- no copying; enables procedures to return multiple values
• Disadvantages: Aliasing problem -- unexpected side effects; harder to optimize
13.3 Call by Value-Result (Copy-in Copy-out):
CALL BY VALUE-RESULT
Phase 1 -- Copy-in: At the START of the call, the actual parameter's value is COPIED into the formal parameter.
Phase 2 -- Procedure executes on its LOCAL COPY.
Phase 3 -- Copy-out: At the END of the call, the final value of the formal parameter is COPIED BACK to the actual
parameter location.
The caller's variable is affected -- but only once, at the end, not during execution.
• Used in: Ada (IN OUT mode), some Fortran implementations
• Differs from call by reference when: aliasing occurs -- changes are batched and written back at
end
• Advantages: Suitable for remote procedure calls where continuous pointer access is impossible
13.4 Call by Name:
CALL BY NAME
The actual parameter expression (not its value, not its address) is TEXTUALLY SUBSTITUTED for the formal
parameter.
The expression is RE-EVALUATED every time the formal parameter is accessed inside the procedure.
Introduced in ALGOL 60. Implemented using THUNKS (closures capturing the expression and its environment).
• Used in: ALGOL 60 (historically); Haskell's lazy evaluation is similar
• Implementation: Thunk -- a parameterless closure representing the unevaluated argument
• Disadvantages: Very expensive (re-evaluates expression on each access); non-intuitive aliasing
behavior

14. STORAGE / MEMORY ALLOCATION STRATEGIES
14.1 Static Allocation:
STATIC ALLOCATION
Memory is allocated at COMPILE TIME. The compiler assigns a FIXED, permanent address to each variable.
No allocation or deallocation occurs at runtime -- variables exist for the ENTIRE duration of program execution.
• Global variables (declared outside any function)
• Static local variables (declared with 'static' keyword in C)
• String literals, global constants
• In Fortran 77: ALL local variables (no recursion support)
• Advantages: Zero runtime overhead; addresses known at compile time; simplest to implement
• Limitations: Cannot support recursion; all data sizes must be known at compile time
14.2 Stack-Based (Dynamic) Allocation:
STACK ALLOCATION
Activation records are allocated on the RUNTIME STACK using LIFO discipline.
Memory is automatically allocated when a procedure is called and freed when it returns.
Fully supports recursion -- each recursive call gets its own separate activation record with its own locals.
• Advantages: Supports recursion; efficient O(1) allocation/deallocation; no fragmentation
• Limitations: Size of each activation record must be known at compile time; lifetime tied to
procedure activation
14.3 Heap Allocation:
HEAP ALLOCATION
The heap is a large pool of free memory that programs can request and release in ARBITRARY ORDER at runtime.
Allocation: program calls malloc(size) / new -- heap manager finds a free block of sufficient size.
In managed languages (Java, Python, C#): a Garbage Collector automatically reclaims unreachable objects.

Answer:
Semantic Actions are code fragments embedded within grammar productions that define the meaning

Q.5(b) [7 marks] - Different symbol table organizations
Answer:
The five major organizations are:
• 1. Linear List: O(n) lookup. Used only for tiny programs.
• 2. Hash Table (chaining): O(1) average. Industry standard (GCC, LLVM, JVM).
• 3. BST (balanced): O(log n). Produces sorted output for cross-reference tables.
• 4. Self-Organizing List: Adapts to hot identifiers via move-to-front strategy.
• 5. Stack of Hash Tables: For block-structured languages. Correctly implements lexical scoping.

Q.6(b) [7 marks] - What is an activation record? When is it set up?
Answer:
An Activation Record is a block of memory allocated on the runtime stack to hold all information
required for a single invocation of a procedure or function. Every time a procedure is called, a fresh
activation record is created. Every time the procedure returns, its activation record is destroyed.
Setup Sequence:
• Caller's actions: Evaluates actual parameters; pushes them on stack; executes CALL instruction
(saves return address, jumps to callee).
• Callee's prologue: Saves old FP as control link; sets FP = SP; decrements SP by frame size;
saves callee-saved registers.
• On return (epilogue): Stores return value; restores saved registers; restores SP = FP; restores
FP = control_link; executes RETURN.

Q.5(b) [4 marks] - Explain inherited and synthesized attributes
Synthesized Attribute: An attribute whose value is computed from the attributes of its CHILDREN in
the parse tree. Information flows UPWARD from leaves to root. Example: E.val = E1.val + T.val (value
of E synthesized from children E1 and T).
Inherited Attribute: An attribute whose value is computed from the attributes of its PARENT or LEFT
SIBLINGS in the parse tree. Information flows DOWNWARD. Example: L.in = T.type in D -> T L -- the
type is inherited by the identifier list from the type specifier.

Q.5(a) [6 marks] - Sources of error in a compiler
Lexical Errors:
Invalid characters, unterminated strings, malformed numeric literals. Recovery: skip to next delimiter.
Syntax Errors:
Missing semicolons, mismatched parentheses, invalid token in expression. Recovery: error productions,

Semantic Errors:

Logical Errors:
Program compiles and runs but produces incorrect results. CANNOT be caught by compiler.
Runtime Errors:
Division by zero, null pointer dereference, stack overflow, array index out of bounds.
Q.5(b) [8 marks] - Data structure for symbol table in block structured language
The Stack of Symbol Tables Design:
One hash table per active scope, maintained in a stack. Entering a block pushes a new table; exiting
pops it. Lookup starts at top (innermost scope) and works downward.
int a = 1; // Global scope (table 0): {a:int}
void f(int x) { // ENTER SCOPE -> push table 1: {x:int}
float a = 3.14; // table 1: {x:int, a:float} -- shadows global a
{ // ENTER SCOPE -> push table 2
int b = 99; // table 2: {b:int} -- shadows global b
int c = a; // lookup(a): table2->no, table1->YES (float a)
} // EXIT SCOPE -> pop table 2
} // EXIT SCOPE -> pop table 1
Q.6(a) [7 marks] - Memory allocation in procedure call and return
Memory allocation in procedure call/return uses a RUNTIME STACK with LIFO discipline. On call: push
activation record. On return: pop activation record. Each recursive call gets its own frame.
Q.7(a) [7 marks] - Different loop optimization techniques
Code Motion:
Move computations that produce the same value every iteration outside the loop. Example: while(i <
a*b) -> t=a*b; while(i < t).
Induction Variable Elimination:
Replace expensive multiplications with cheaper additions for loop variables in arithmetic progressions.
Strength Reduction:
Replace expensive operations with cheaper equivalents. Multiply replaced by addition.
Loop Unrolling:
Replicate the loop body multiple times, reducing branch/counter update overhead.
Loop Fusion:
Combine two adjacent loops with the same iteration range into a single loop.

Q.6(b) [7 marks] - Run time storage management for call and return
On Call:
• Evaluate and push actual parameter values
• CALL instruction: push return address, jump to callee
• Callee: push current FP (control link), set FP = SP
• Callee: SP = SP - sizeof(locals + temps) to allocate locals
• Callee saves callee-saved registers; begins execution
On Return:
• Place return value at designated location (or register EAX/RAX)
• Restore callee-saved registers
• Restore SP = FP; Restore FP = control_link (old frame pointer)
• RETURN instruction: pop return address, jump to it
• Caller reads return value; pops actual parameters if caller's responsibility
